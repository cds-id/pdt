WhatsApp Analytics — Architecture Approaches

3 approaches for integrating WhatsApp messaging into PDT

A

Monolith Extension (Recommended)

Add WhatsApp directly into the existing PDT backend. whatsmeow connections run as managed goroutines alongside existing sync workers. New DB tables for numbers, listeners, messages, files. A new WhatsAppAgent joins the existing 5 agents in the orchestrator. Weaviate client lives in PDT as a service, called by the agent for context-aware queries.

Pros

  • Simplest — reuses existing patterns (services, agents, DB, R2)
  • Shared auth, config, and WebSocket infrastructure
  • Single deployment unit
  • WhatsApp agent can cross-reference Jira/Git data natively

Cons

  • whatsmeow connections add memory/CPU to main process
  • If WA connection crashes, could affect main server
  • Harder to scale WA connections independently
B

Sidecar Process

WhatsApp connections run in a separate Go binary (sidecar) that handles all whatsmeow sessions and message ingestion. Sidecar writes messages to the shared MySQL + R2 and pushes embeddings to Weaviate. PDT backend reads from DB/Weaviate — the WhatsApp Agent queries data but doesn't manage connections.

┌──────────────────────┐    ┌──────────────────────┐
│   PDT Backend (Go)   │    │  WA Sidecar (Go)     │
│                      │    │                      │
│  ┌────────────────┐  │    │  ┌────────────────┐  │
│  │ Existing + WA  │  │    │  │ Connection Mgr │  │
│  │ Agents (6)     │  │    │  │ (whatsmeow)    │  │
│  └───────┬────────┘  │    │  └───────┬────────┘  │
│          │           │    │          │           │
│  ┌───────┴────────┐  │    │  ┌───────┴────────┐  │
│  │ Weaviate Client│  │    │  │ Msg Processor  │  │
│  └───────┬────────┘  │    │  └───────┬────────┘  │
└──────────┼───────────┘    └──────────┼───────────┘
           │                           │
    ┌──────┴───────────────────────────┴──────┐
    │          MySQL + R2 + Weaviate           │
    └─────────────────────────────────────────┘

Pros

  • Process isolation — WA crash doesn't kill PDT
  • Can scale/restart WA independently
  • Clean separation of concerns

Cons

  • Two binaries to build, deploy, and monitor
  • Sending messages requires IPC (HTTP/gRPC) between PDT → sidecar
  • More operational complexity for a single-user app
C

Microservice Split

Full microservice: dedicated WhatsApp service with its own API, separate from PDT. Communicates via REST/gRPC. Owns its own message storage. PDT queries it for analytics data.

Pros

  • Maximum isolation and independence
  • Could be reused by other projects

Cons

  • Overkill for a personal tool — significant overhead
  • Data duplication or complex cross-service queries
  • Loses the benefit of agent cross-referencing Jira/Git
