Section 5 (Updated): Weaviate + Gemini Embeddings

Using Google Gemini as vectorizer instead of local transformers

Updated Docker Setup
# docker-compose.yml — simpler, no transformer sidecar

services:
  weaviate:
    image: semitechnologies/weaviate:latest
    ports:
      - "8081:8080"
      - "50051:50051"
    environment:
      QUERY_DEFAULTS_LIMIT: 25
      AUTHENTICATION_ANONYMOUS_ACCESS_ENABLED: true
      PERSISTENCE_DATA_PATH: /var/lib/weaviate
      DEFAULT_VECTORIZER_MODULE: text2vec-palm
      ENABLE_MODULES: text2vec-palm
      CLUSTER_HOSTNAME: node1
    volumes:
      - weaviate_data:/var/lib/weaviate

# No transformer container needed!
# Gemini API key configured in Weaviate collection schema
# or passed via request headers

What Changed

Removed

Local t2v-transformers container — no more CPU/GPU-heavy inference sidecar

Added

GEMINI_API_KEY env var in PDT config. Passed to Weaviate via request headers on each batch upsert and query.

Better Quality

Gemini embeddings are higher quality than local MiniLM, especially for multilingual (Indonesian + English) content.

Rate Limiting

Embedding worker respects Gemini API rate limits. Batch upserts reduce API calls. Backfill job uses slower pacing to stay within quota.

Everything Else Unchanged

Resilience model stays the same: MySQL is source of truth, async embedding via Go channel, idempotent upserts, graceful degradation if Weaviate/Gemini is down, backfill support.
